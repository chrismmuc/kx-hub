## Key Design Decisions (Zusammenfassung)

1. **Filter am Anfang, nicht am Ende** — Sources werden vor dem Generator klassifiziert (nicht aus der privaten Summary extrahiert). Genauer, auch wenn ~2× Compute.

2. **ADK Agent für Filter + Research** — Kein Regelwerk (Allowlist/Denylist/Schwellwerte). Ein ADK Agent auf Vertex AI Agent Engine übernimmt beides: Source-Filterung + Hot News Recherche via `google_search`. Kosten-Delta vs. explizitem LLM-Ansatz: ~$0.25/Monat — akzeptabel für deutlich weniger Wartungsaufwand.

3. **Brevo für E-Mail** — Free Tier (300/Tag) reicht für den Start. Bei Wachstum: Brevo Starter ($9/Monat).

4. **Eigener Newsletter ≠ Extrakt der privaten Summary** — Unterschiedliche Pipeline (gefilterter Input, anderer Prompt, englisch, externer Ton). Qualitativ besser als Post-Processing.

5. **Agent Input/Output explizit spezifiziert:**
   - Input: Sources als JSON-serialisierte User-Message (einfachste Lösung für Agent Engine)
   - Output: Pydantic-Schema (`CurationResult`) + Regex-Extraktion mit Fallback bei Parse-Fehler
   - Deployment: `deploy_agent.py` einmalig (`ReasoningEngine.create()`), wöchentlich nur `engine.query()`

6. **Dry-Run vor Live:** Phase-2-MVP läuft N Wochen ohne Versand — Output in Firestore `newsletter_drafts`, manuelles Review, Prompt-Iteration. Erst dann Delivery freischalten.

7. **`kx_search` als zweites Agent-Tool (nach Dry-Run):** HTTP-Call auf MCP Cloud Run Server (einzige praktikable Option auf Vertex AI Agent Engine). Benötigt `roles/run.invoker` IAM für den Agent-Service-Account. MVP startet ohne `kx_search` — erst ergänzen wenn Kuratierungsqualität grundsätzlich stimmt.
